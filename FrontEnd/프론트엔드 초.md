1. 브라우저 렌더링 과정  
(1) HTML/CSS 파싱  
- HTML → DOM 트리 생성  
- CSS → CSSOM 트리 생성  
- <script> 실행 시 HTML 파싱 중단  

(2) Render Tree 생성  
- DOM + CSSOM 결합  
- 화면에 표시되는 요소만 Render Tree에 포함  

(3) Layout(Reflow)  
- 요소의 위치·크기 계산  

(4) Paint  
- 요소를 픽셀 단위로 그리기  

(5) Composite  
- 레이어 단위 합성 → 최종 화면 출력  

2. URL 입력 → 페이지 표시 과정  
(1) URL 입력  
- 브라우저가 URL을 프로토콜 / 도메인 / 포트 / 경로로 파싱  

(2) DNS 조회 순서  
- 브라우저 캐시  
- OS 캐시  
- 라우터  
- ISP DNS  
- 최종 네임서버  
→ IP 획득  

(3) TCP 3-way handshake  
- 클라이언트 ↔ 서버 연결 성립  

(4) 서버로 HTTP 요청 전송  
- Header / Method / Body(Payload)  

(5) 서버 동작(Nginx/Apache → WAS)  
- 웹 서버에서 라우팅  
- WAS에서 비즈니스 로직 수행  
- DB 조회  
- 결과(JSON/HTML) 생성  

(6) 서버 응답 반환  

(7) 브라우저 렌더링 진행  

3. DNS란?  
- 도메인 이름을 IP 주소로 변환하는 시스템  
예: google.com → 142.251.xxx.xx  

4. URI vs URL  
URI: 자원의 식별(더 큰 개념)  
URL: 자원의 위치(URI의 하위)  
예시  
URI: /users/123  
URL: https://example.com/users/123  

5. RESTful API 규칙  
- 슬래시(/)로 계층 표현  
- 마지막 / 금지  
- 명사 기반 설계  
- 하이픈(-) 허용 / 언더바(_) 지양  
- 파일 확장자 사용 X  
- HTTP 메서드로 행동 표현(GET/POST/PUT/PATCH/DELETE)  
- 응답은 JSON 사용  

6. HTTP Method 정리  
GET: 조회  
POST: 생성  
PUT: 전체 수정  
PATCH: 부분 수정  
DELETE: 삭제  

7. Payload vs Header  
Header: 요청·응답의 메타데이터  
- Authorization  
- Content-Type  
- Cookie  
Payload(body): 실제 데이터(JSON 등)  

8. CORS  
(1) CORS란?  
- 브라우저의 Same-Origin Policy(SOP) 때문에 다른 출처 리소스 요청 제한  

(2) 해결 방법  
- 서버에서 CORS 헤더 설정 (Access-Control-Allow-Origin 등)  
- 프록시 서버 사용  
- 브라우저 확장 프로그램(개발 환경 한정)  

9. JWT vs Session  
Session  
- 서버가 세션 저장  
- 클라이언트는 Cookie로 session ID만 전달  
- 서버 메모리/Redis 필요  

JWT  
- 토큰 내부에 정보 자체 포함  
- 서버는 검증만 → Stateless  
단점: 탈취 시 위험, 강제 만료 처리 어려움  

10. OAuth 전체 흐름  
1) Request Token 요청  
2) 서비스 제공자가 Request Token 발급  
3) 사용자 로그인·권한 승인  
4) Access Token 발급  
5) Access Token 기반 API 접근  

11. HTTP vs HTTPS  
HTTP: 암호화 X / 보안 낮음 / 인증서 없음  
HTTPS: SSL/TLS 암호화 O / 보안 높음 / 인증서 필요  

12. localStorage / sessionStorage / cookie  
localStorage: 브라우저 종료 후에도 유지(약 5MB) / 반영구 저장  
sessionStorage: 탭 닫으면 삭제(약 5MB) / 탭 독립  
cookie: 만료 기반 / 약 4KB / 요청 시 자동 전송  

13. Semantic Markup  
- 의미 있는 HTML 태그 사용  
- 예: header, footer, nav  
- SEO 유리  
- 스크린 리더 접근성 향상(A11y)  

14. attributes vs properties  
attributes: HTML 문서에 작성된 고정 값  
properties: DOM 객체의 동적 프로퍼티  

15. SPA / CSR / SSR  
SPA  
- 최초 로드 이후 JS로 페이지 전환  
- 장점: 빠른 화면 전환  
- 단점: SEO 약함, JS 의존도 높음  

CSR  
- 브라우저에서 렌더링  
- 초기 로드 느림  

SSR  
- 서버에서 HTML 생성 후 전달  
- 초기 렌더링 빠름  
- SEO 유리  

16. Webpack / Babel / Polyfill  
Webpack: 모듈 번들러(여러 JS/CSS → 하나)  
Babel: 최신 JS → 구형 브라우저 호환 코드로 변환  
Polyfill: 브라우저가 지원하지 않는 기능을 미리 구현  

17. 페이지 로딩 속도 개선 방법  
- 이미지 최적화  
- Lazy Loading  
- 코드 스플리팅  
- 캐싱 활용  
- CDN 사용  
- 불필요한 JS 제거  
- 파일 압축(minify)  

18. 테스트 종류  
Unit Test: 함수/모듈 단위 테스트  
Integration Test: 여러 모듈을 묶어 테스트  
E2E Test: UI → API까지 실제 사용자 플로우 테스트  

19. MVC / MVVM  
MVC  
- Model: 데이터  
- View: UI  
- Controller: 요청 처리·로직  

MVVM  
- Model: 데이터  
- View: UI  
- ViewModel: 상태·로직 관리  
- 양방향 바인딩 용이  

20. 배포 환경  
- S3 정적 호스팅 가능  
- CloudFront CDN  
- GitHub Actions 등으로 CI/CD 가능  
- 무중단 배포: Blue-Green 방식  

21. CI/CD 전체 개념  
CI (Continuous Integration)  
- 코드 push 시 자동 빌드·테스트  
- 코드 품질 보증  
- 팀 협업 효율 증가  

CD (Continuous Delivery/Deployment)  
- 자동 배포  
- staging → production 자동화 가능
